# Android-приложения по билетам к экзамену

В этом репозитории собраны готовые исходники **25 простых Android-приложений на Java** —
по одному на каждый билет экзамена. Всё написано на простом уровне только стандартный Android SDK,
без сторонних библиотек (Retrofit, Room, Glide, ViewBinding и т.п.), без RecyclerView,
без MVVM и прочих архитектурных паттернов. Один экран = одна `Activity`, `findViewById` +
слушатели, данные в `SharedPreferences` (JSON через встроенный `org.json`).

---

## Как пользоваться на экзамене

1. Создай новый проект в Android Studio: **Empty Views Activity**, язык **Java**.
2. Открой нужную папку приложения (например `10_Калькулятор_чаевых`).
3. Скопируй файлы по назначению:
   - `MainActivity.java` → `app/src/main/java/<твой.пакет>/MainActivity.java` (заменить содержимое)
   - остальные `*.java` (если есть) → в ту же папку пакета
   - `activity_*.xml`, `item_*.xml` → `app/src/main/res/layout/`
   - `AndroidManifest.xml` → `app/src/main/AndroidManifest.xml` (заменить или сверить)
4. **Поправь имя пакета руками** (это главный ручной шаг):
   - первая строка каждого `.java`: `package com.example.xxx;` → на пакет твоего проекта;
   - в `AndroidManifest.xml` атрибут `package="com.example.xxx"` — привести к тому же пакету
     (в новых проектах пакет задаётся через `namespace` в `build.gradle` — тогда атрибут
     в манифесте можно убрать, главное чтобы `.java` лежали в правильной папке пакета).
5. Добавь разрешения из манифеста приложения, если они там есть (см. таблицу ниже).
6. Запусти на эмуляторе/телефоне.

В **каждой** папке есть `README.txt` с точной инструкцией именно для этого приложения
(куда что копировать, что поменять, какие разрешения, нюансы, минимальный SDK).

## Разрешения (кратко)

Большинству приложений разрешения **не нужны**. Исключения:

| Приложение         | Разрешение                                                  | Зачем                                               |
| ------------------ | ----------------------------------------------------------- | --------------------------------------------------- |
| 02, 12             | `INTERNET`                                                  | сетевые запросы к API                               |
| 12                 | `ACCESS_FINE/COARSE_LOCATION`                               | кнопка «моё местоположение» (запрос в рантайме)     |
| 09, 13, 18, 21, 22 | `POST_NOTIFICATIONS`                                        | уведомления (нужно с Android 13, запрос в рантайме) |
| 07                 | камера через Intent — **отдельное разрешение не требуется** | съёмка делает системная камера                      |

## Важные оговорки (чтобы не было сюрпризов)

- **Сеть только в фоне.** Приложения 2 и 12 делают запросы в отдельном `Thread` и обновляют
  экран через `runOnUiThread` — иначе Android падает с `NetworkOnMainThreadException`.
- **Уведомления** (9, 13, 18, 21, 22) показываются в момент действия — это демонстрация
  механизма. Чтобы они приходили строго по расписанию в фоне, нужен `AlarmManager`/`WorkManager`
  (в учебных целях не делается, чтобы не усложнять).
- **QR (№7)** — настоящий стандартный QR требует библиотеки (ZXing), которая запрещена
  правилами «без сторонних библиотек». Поэтому генерация — это визуальная имитация через
  `Canvas`, а сканер открывает системную камеру. Подробности в `07_.../README.txt`.
- **Погода (№12)** — «карта осадков» заменена на количество осадков в мм по дням (полноценная
  карта требует картографического SDK).
- **Графики** (15, 19, 20) нарисованы вручную на `Bitmap`/`Canvas`, без chart-библиотек.
- **Тема.** В манифестах указана `Theme.AppCompat.Light.NoActionBar` — она есть в стандартном
  проекте Android Studio. Если тема другая — просто оставь ту, что сгенерировала студия.
- **minSdk:** для приложений с уведомлениями и геолокацией рекомендуется **23+**, остальным
  подойдёт 21+. Проверки версий (`Build.VERSION.SDK_INT`) в коде уже есть.

## Стиль кода

Во всех приложениях выдержан единый уровень комментариев на русском: пояснение к каждому
методу и к каждому логическому блоку, короткие inline-комментарии на неочевидных местах.
Имена переменных простые (`a`, `sum`, `list`, `o`), без лишних абстракций — чтобы код было
легко прочитать и воспроизвести на экзамене.
